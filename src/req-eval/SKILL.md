# 短途旅行 · "需求 → 评价指标"映射库的存储与优化

本技能管理的记忆是一套**评估规则库**:它记录"**某类用户需求,应当用什么评价指标、如何判定**
规划结果是否满足"。记忆单元是**需求→指标的映射**,**不是**具体的旅行地点、景点或某用户的口味事实。
随着 agent 完成更多规划任务,这套映射被不断校准、归并、剪枝,使评估越来越准、越来越省。

## 技能文件清单(存储位置)

本技能文件夹内的文件,即各类资料的**固定存储位置**:

| 文件 | 角色 | 何时读 | 何时写 |
|------|------|--------|--------|
| `SKILL.md` | 技能说明(本文件) | 任务开始 | 一般不改 |
| `questionnaire.md` | 树形问卷 + 需求表 schema(需求侧来源) | 采集需求前 | 问卷字段变更时 |
| `questionnaire.html` | 可视化问卷,作为 iframe 嵌入界面采集需求 | 需要可视化交互采集时 | 问卷字段变更时同步 |
| `mappings.json` | **「需求→评价指标」映射库(持久存储)** | **每次评估前读取** | **每次评估后按需校准/泛化/剪枝写回** |

> 关键:**已有的需求→评估指标全部存在 `mappings.json`**。它是本技能唯一的"活"数据文件——
> agent 评估前从这里取规则,评估后把改进写回这里。其余文件相对静态。

---

## 1. 何时使用

- 需要为一次短途旅行规划结果**建立评估标准 / 打分**。
- 需要把《旅行需求表》中的某条需求**转化为可判定的评价指标**。
- 复用并改进历史上沉淀的"需求→指标"评估规则。

> 区分:具体的"用户忌辣""这次去 XX 古镇"属于**事实型记忆**,不归本技能管理;
> 本技能只管"**忌辣这类饮食禁忌 → 应检查餐厅是否全部规避该禁忌(硬过滤/命中率)**"这种**规则型映射**。

---

## 2. 记忆单元:一条"需求→指标"映射

每条记忆描述**一类需求**到**一个评价指标**的对应,字段如下:

| 字段 | 含义 | 示例 |
|------|------|------|
| `requirement_key` | 对应问卷/需求表的字段或取值 | `preferences.diet = 忌辣` |
| `metric` | 评价指标名 | 餐厅饮食禁忌规避率 |
| `type` | 硬约束 / 硬过滤 / 软偏好 | 硬过滤 |
| `judge_method` | 判定方式:程序化 / LLM+rubric | 程序化 |
| `pass_criteria` | 通过标准 / 打分标尺 | 所有餐饮项均不含忌口项 = 通过 |
| `evidence` | 判定需引用的行程字段 | itinerary[].meals[].cuisine |
| `notes` | 校准经验、坑、阈值来源 | "辣度模糊时按保守判定为不通过" |

> 这套映射的**初始种子**就是 `questionnaire.md` 第 6 节(选项→规划约束映射)和第 7 节(评估衔接)。
> 本技能负责把它从一张静态表,养成一个**可积累、可优化**的记忆库。

---

## 3. 三层指标结构(分类存储)

按"判定可信度"把映射分三层存,**不要混成一锅 LLM 打分**:

| 层级 | 典型需求来源 | 判定方式 | 记忆要点 |
|------|--------------|----------|----------|
| **规则层(最可信)** | 预算、必去清单、忌口、无障碍、通勤上限、时间冲突 | 程序化/确定性 | 记清**阈值与加总逻辑** |
| **常识层** | 空间-时间合理性、节奏、常识违背 | LLM + 明确 rubric | 记清**rubric 条目** |
| **体验层(最主观)** | 主题契合、行程吸引力、实用性 | LLM 独立调用 + 引用证据 | 记清**需引用的证据** |

> 每条映射记忆都应标明它属于哪一层,决定它走代码校验还是 LLM 评判。

---

## 4. 存储与优化规则(核心)

**存储位置固定为本技能文件夹内的 `mappings.json`**(结构见该文件 `_meta` 与 `mappings[]`,
每条含第 2 节的 7 字段 + `layer`)。所有读取与更新都针对此文件。执行以下优化纪律:

1. **先查后写(去重)**:写入新映射前检索同 `requirement_key` 的记忆。已存在 → **更新/补充**,
   绝不新建重复映射。

2. **冲突即收敛**:同一需求出现两条矛盾指标(如忌口一次按"硬过滤"一次按"软偏好")→
   收敛为一条,保留更合理者,并在 `notes` 记录定夺理由。不留并存矛盾。

3. **校准而非堆叠**:每次评估后若发现指标失准(阈值过松/过紧、误判),
   **修改既有映射的 `pass_criteria`/阈值**,而不是新增一条。记录"何时、因何校准"。

4. **泛化(优化关键)**:多条具体映射呈现同一模式时,**抽象成一条通用映射**并删除冗余。
   - 例:"忌辣→规避""清真→规避""素食→规避"可泛化为
     "任一 `diet` 禁忌 → 餐饮项规避率(硬过滤)"一条规则。

5. **剪枝**:长期**区分度低**(几乎总通过/总不通过)或**判定不可靠**的指标 →
   降级或删除,避免评估噪声。剪枝要 `log` 说明删了什么,防止"看起来覆盖全了"的假象。

6. **标层不混判**:每条映射用 `layer` 字段标明规则层/常识层/体验层,据此决定走程序化还是 LLM 评判。

7. **保持与问卷同步**:问卷字段增删时,同步增删对应映射;
   `requirement_key` 必须能对上 `questionnaire.md` 的实际字段,否则映射成孤儿。

8. **关联链接**:相关映射在 `notes` 里用 `[[requirement_key]]` 互引(如"无障碍"与"避免爬山"相互关联)。

9. **每次写回都更新 `_meta`**:修改 `mappings.json` 后递增 `version`、更新 `last_calibrated`,
   便于追踪库的演进(无法取系统时间时,由调用方传入日期)。

---

## 5. 可视化问卷的嵌入与回收(采集需求)

需要在交互中可视化采集需求时,使用 `questionnaire.html`:

1. **嵌入**:把 `questionnaire.html` 作为 `iframe` 插入 agent 的可视化界面。
   它是自包含单文件(内联 CSS/JS),无外部依赖,可直接用 `srcdoc` 或文件 URL 加载。
   ```html
   <iframe src="req-eval/questionnaire.html" style="width:100%;height:640px;border:0"></iframe>
   ```
2. **分支**:问卷内置树形联动——勾选「带儿童/带老人/带宠物」「两天一夜」「美食/户外」等
   会自动展开对应子模块(逻辑与 `questionnaire.md` 一致)。
3. **回收结果**:用户点「生成需求表」后,iframe 通过 `postMessage` 向父窗口发送结构化需求表:
   ```js
   window.addEventListener('message', (e) => {
     if (e.data?.type === 'trip-req-eval:submit') {
       const reqForm = e.data.payload;   // 即《旅行需求表》JSON,字段同 questionnaire.md 第 5 节
       // → 送入第 6 节的评估流程
     }
   });
   ```
4. **同步约束**:若改动了 `questionnaire.md` 的字段,必须同步改 `questionnaire.html` 与 `mappings.json`,
   三者的字段名(`requirement_key`)保持一致,否则采集到的需求无法匹配到评价指标。

---

## 6. 应用:读取映射库 → 生成评估方案 → 写回

**读取时机:每次评估某份规划结果之前,先完整读取 `mappings.json`。** 然后:

1. 读取本次《旅行需求表》(来自 HTML 问卷的 `postMessage` 或文本采集),取出所有非默认字段。
2. 对每个字段,在 `mappings.json` 中按 `requirement_key` **匹配**对应的评价指标。
3. 按 `layer` 组装评估方案:规则层走程序化校验,常识/体验层组装成 LLM rubric(独立调用、要求引用证据)。
4. 产出分层评分:硬约束满足度(可复现) + 软偏好覆盖率(承认主观)。

**更新时机:每次评估之后,把本轮暴露的问题按第 4 节规则写回 `mappings.json`**:

- 匹配不到指标的需求字段 → **新增**一条映射(走去重检查)。
- 指标失准(阈值过松/过紧、误判)→ **校准**既有条目的 `pass_criteria`/阈值,不新增。
- 出现可合并的同模式条目 → **泛化**为一条并删冗余。
- 长期低区分度/不可靠的指标 → **剪枝**(并 `log` 说明删了什么)。
- 写回后更新 `_meta.version` / `_meta.last_calibrated`。

---

## 7. 生命周期(养库,不是攒库)

- **新增**:遇到问卷里有、但 `mappings.json` 尚无对应指标的需求 → 新建一条映射。
- **校准/泛化/剪枝**:见第 4 节,这是映射库随经验变好的主路径。
- **同步淘汰**:问卷删除某字段 → 删除 `mappings.json` 中其孤儿映射。
- **体检**:定期检查矛盾、低区分度、长期未命中的映射并清理。

> 注意:本库**不随单次行程增长**。它的规模应趋于稳定(与问卷字段数同量级),
> 增长的是**质量**(阈值更准、判定更可靠),而非条目堆积。
